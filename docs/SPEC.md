# Техническое задание — «Бизнес-Симулятор: Управление и Экономика»

> Однопользовательская экономическая симуляция с ИИ-конкурентами  
> Аналог MESE (Management and Economic Simulation Exercise)  
> Версия 3.0 | Март 2026

---

## 1. Общие сведения

### 1.1. Название проекта

**«Бизнес-Симулятор: Управление и Экономика»** (рабочее название: BizSim).

### 1.2. Назначение и цели

Веб-приложение представляет собой автономную однопользовательскую экономическую симуляцию, аналогичную MESE (Management and Economic Simulation Exercise), полностью на русском языке. Игрок управляет своей компанией и конкурирует с ИИ-оппонентами на едином рынке, принимая стратегические решения в области ценообразования, производства, маркетинга и инвестиций. Игра полностью автономна — не требует других игроков, организатора или подключения к серверу.

**Основные цели:**

- Самостоятельное обучение основам предпринимательства и экономики через практический опыт
- Развитие навыков принятия управленческих решений в условиях конкуренции с ИИ
- Возможность играть в любое время без ожидания других участников
- Настраиваемая сложность для игроков разного уровня подготовки

### 1.3. Целевая аудитория

- Студенты экономических и управленческих специальностей
- Учащиеся старших классов (14–18 лет), изучающие экономику
- Все, кто хочет попрактиковаться в управлении виртуальным бизнесом в свободном режиме

### 1.4. Язык интерфейса

Русский — основной язык. Весь интерфейс, отчёты, подсказки, сообщения об ошибках и документация должны быть на русском языке. Архитектура должна предусматривать возможность добавления других языков (i18n).

### 1.5. Ключевое отличие от классического MESE

Классический MESE требует нескольких команд и организатора. Данное приложение полностью автономно:

- Игрок играет один против ИИ-компаний (ботов)
- Не нужна регистрация, лобби, ожидание других игроков
- Сложность ИИ выбирается перед началом игры
- Игра может работать полностью на клиенте (SPA) без обязательного серверного бэкенда
- Сохранение прогресса в локальном хранилище браузера (localStorage / IndexedDB)

## 2. Концепция игры

### 2.1. Общее описание

Игрок управляет одной виртуальной компанией, производящей условный товар — «Эко-Ручка» (аналог Eco-Pen в MESE). На рынке одновременно действуют от 2 до 7 ИИ-компаний. Симуляция проходит походово: в каждом периоде (квартале) игрок принимает 5 ключевых решений, ИИ-оппоненты одновременно принимают свои, и система рассчитывает результаты.

### 2.2. Настройка новой игры

Перед началом игры пользователь задаёт параметры сессии на экране «Новая игра»:

| Параметр                  | Описание                                           | Значение                                      |
| ------------------------- | -------------------------------------------------- | --------------------------------------------- |
| **Сложность ИИ**          | Уровень интеллекта ИИ-конкурентов (см. раздел 2.4) | Новичок / Средний / Эксперт / Мастер          |
| **Кол-во ИИ-конкурентов** | Сколько компьютерных компаний на рынке             | 2–7 (по умолчанию 4)                          |
| **Кол-во периодов**       | Длительность игры в раундах                        | 8–24 (по умолчанию 12)                        |
| **Название компании**     | Имя компании игрока (опционально)                  | Текстовое поле                                |
| **Рыночные условия**      | Предустановленные сценарии экономики               | Стабильный / Растущий / Кризисный / Случайный |

### 2.3. Основные решения игрока

В каждом периоде игрок задаёт 5 параметров:

| №   | Параметр                   | Описание                                                                                                 |
| --- | -------------------------- | -------------------------------------------------------------------------------------------------------- |
| 1   | **Цена продукции**         | Розничная цена за единицу «Эко-Ручки» в условных денежных единицах (УДЕ). Влияет на спрос и выручку.     |
| 2   | **Объём производства**     | Количество единиц продукции на следующий период. Влияет на себестоимость и запасы.                       |
| 3   | **Маркетинг**              | Бюджет на рекламу и продвижение. Увеличивает спрос с убывающей отдачей.                                  |
| 4   | **Капитальные инвестиции** | Вложения в оборудование. Повышают производительность и снижают себестоимость в долгосрочной перспективе. |
| 5   | **НИОКР (R&D)**            | Инвестиции в разработки. Улучшают качество продукции и увеличивают спрос.                                |

### 2.4. Уровни сложности ИИ

Каждый уровень определяет поведение ИИ-компаний — от примитивных стратегий до агрессивной адаптации:

| Уровень     | Поведение ИИ                                                                                                                                                                      |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Новичок** | ИИ действует по фиксированным правилам с большим случайным разбросом. Не адаптируется к действиям игрока. Иногда принимает неоптимальные решения. Подходит для изучения механики. |
| **Средний** | ИИ использует базовые эвристики (снижает цену при низких продажах, наращивает маркетинг при падении доли). Умеренный случайный разброс. Частично реагирует на рынок.              |
| **Эксперт** | ИИ анализирует действия игрока и рыночные тренды. Применяет стратегии: лидерство по издержкам, дифференциация, агрессивный маркетинг. Минимальный разброс.                        |
| **Мастер**  | ИИ использует оптимальные стратегии с учётом долгосрочного планирования. Активно контратакует игрока: демпинг, переинвестирование в R&D. Нулевой разброс. Сложно обыграть.        |

Разные ИИ-компании на одном рынке могут иметь разные «характеры» (осторожный, агрессивный, сбалансированный) для разнообразия игрового опыта. При уровнях «Эксперт» и «Мастер» ИИ должны применять разные стратегии, чтобы игроку не хватило одной «универсальной тактики».

### 2.5. Индекс эффективности (MPI)

Общий результат определяется Индексом эффективности управления (MPI — Management Performance Index), рассчитываемым на основе 6 факторов:

1. Нераспределённая прибыль (основной фактор)
2. Потенциал спроса
3. Потенциал предложения
4. Производительность
5. Доля рынка
6. Рост (динамика показателей)

Побеждает игрок, набравший наивысший MPI по итогам всех периодов среди всех компаний (своей и ИИ).

### 2.6. Сохранение и продолжение

- Автосохранение состояния игры после каждого периода в localStorage / IndexedDB
- Поддержка нескольких сохранённых игр (слоты сохранения)
- Экспорт/импорт сохранения в JSON-файл (для переноса между устройствами)
- Возможность закрыть браузер и продолжить позже

## 3. Функциональные требования

### 3.1. Главный экран

- Кнопка «Новая игра» — открывает экран настройки (см. раздел 2.2)
- Кнопка «Продолжить» — загрузка последнего сохранения
- Список сохранённых игр с краткой информацией (период, сложность, MPI)
- Кнопка «Как играть» — обучение/правила
- Таблица рекордов (лучшие MPI игрока по уровням сложности)

### 3.2. Игровой экран

#### 3.2.1. Панель принятия решений

- Форма ввода 5 параметров с ползунками и числовыми полями
- Валидация допустимых диапазонов значений с подсказками «рекомендованный диапазон»
- Кнопка «Завершить период» — мгновенный расчёт результатов (без ожидания других игроков)
- Отображение текущего периода и оставшихся периодов
- Краткая сводка показателей прошлого периода (прибыль, продажи, доля рынка, место в рейтинге)

#### 3.2.2. Отчёты и аналитика

После каждого периода игрок получает:

**Отчёт компании:**

- Отчёт о прибылях и убытках (выручка, себестоимость, расходы, налоги, чистая прибыль)
- Балансовый отчёт (активы, обязательства, капитал)
- Остатки продукции, объём продаж, объём производства

**Отраслевой отчёт:**

- Средние показатели по рынку (средняя цена, общие продажи, рыночная доля каждой компании включая ИИ)
- Индекс потребительского доверия и макроэкономические тренды

**Рейтинг:**

- Текущий MPI игрока и каждой ИИ-компании
- Место игрока в общем рейтинге с выделением цветом

#### 3.2.3. Подсказки и обучение

- Обучающий тур при первом запуске (interactive tutorial) с пошаговыми пояснениями
- Тултипы к каждому параметру: что означает, на что влияет, рекомендации
- Всплывающие подсказки при критических ситуациях (например: «Ваши запасы на складе растут — рассмотрите снижение цены») — можно отключить в настройках
- Справочник по экономическим понятиям внутри приложения

### 3.3. Экран итогов игры

- Финальный рейтинг всех компаний по MPI
- Подробная статистика: графики динамики за всю игру (прибыль, доля рынка, MPI)
- Сравнение с ИИ: где игрок опережал/отставал
- Кнопка «Новая игра» и «На главную»
- Экспорт результатов в PDF

### 3.4. Визуализация и графики

- Линейные графики динамики (прибыль, выручка, доля рынка, MPI) — компания игрока выделена цветом на фоне ИИ
- Столбчатые диаграммы сравнения с ИИ-компаниями
- Круговые диаграммы доли рынка
- Таблицы с финансовыми показателями всех компаний

## 4. ИИ-оппоненты (подробное описание)

### 4.1. Архитектура ИИ

Каждая ИИ-компания — это объект, который на входе получает текущее состояние рынка и свои финансовые показатели, а на выходе возвращает 5 решений на период. ИИ работает полностью на клиенте (JavaScript), без серверных вызовов.

**Модульная структура ИИ:**

- Базовый класс `AIPlayer` с интерфейсом `makeDecisions(marketState, companyState) → Decisions`
- Конкретные стратегии-наследники: `CautiousAI`, `AggressiveAI`, `BalancedAI`, `AdaptiveAI`
- Каждый уровень сложности определяет: набор доступных стратегий, амплитуду случайного разброса, глубину анализа

### 4.2. Характеры ИИ

ИИ-компании на одном рынке должны вести себя по-разному для реалистичности:

- **Осторожный (CautiousAI):** держит средние цены, минимальные риски, медленно наращивает производство
- **Агрессивный (AggressiveAI):** низкие цены, большой маркетинг, стремится к максимальной доле рынка
- **Сбалансированный (BalancedAI):** равномерно инвестирует во все направления, умеренные цены
- **Адаптивный (AdaptiveAI):** анализирует действия игрока и меняет стратегию в ответ (доступен на «Эксперт» и «Мастер»)

При создании игры система автоматически назначает каждой ИИ-компании случайный характер из пула, доступного для выбранного уровня сложности. Имена ИИ-компаний генерируются автоматически из предустановленного списка (например: «Прогресс», «Восток», «Меридиан», «Альфа»).

### 4.3. Балансировка

- На уровне «Новичок» игрок должен стабильно побеждать при разумных решениях
- На уровне «Средний» победа требует осознанной стратегии
- На уровне «Эксперт» победить можно только при глубоком понимании модели
- На уровне «Мастер» победа — вызов, даже опытному игроку нужно найти оптимальную тактику

## 5. Экономическая модель (Simulation Engine)

Ядро симуляции работает полностью на клиенте (JavaScript/TypeScript) и реализует:

- **Расчёт спроса:** зависит от цены (обратная зависимость), маркетинга, НИОКР, макроэкономики
- **Распределение рыночной доли:** на основе конкурентных преимуществ (цена, качество, известность)
- **Себестоимость:** постоянные + переменные затраты, эффект масштаба
- **Амортизация оборудования**, накопление R&D-активов
- **Налогообложение** (настраиваемая ставка)
- **Управление запасами** с расходами на складирование
- **Макроэкономический цикл** (периоды роста/спада) зависит от выбранного сценария
- **Расчёт MPI** по 6 факторам

## 9. Методология разработки (Claude Code)

### 9.1. Принцип: Plan → Tests → Code

Разработка ведётся через Claude Code по методологии TDD (Test-Driven Development). Каждая фича проходит 3 этапа:

**Этап 1 — Plan:**

- Claude Code составляет подробный план реализации фичи в файл `PLAN.md`: структура файлов, интерфейсы, зависимости, краевые случаи
- План утверждается перед началом кодирования

**Этап 2 — Tests (Red):**

- Пишутся юнит-тесты (Vitest) ДО написания реализации
- Тесты описывают ожидаемое поведение модуля (вход/выход, краевые случаи)
- Все тесты должны падать (red) на момент написания

**Этап 3 — Code (Green):**

- Claude Code пишет реализацию, пока все тесты не станут зелёными (green)
- После — рефакторинг (refactor) без поломки тестов
- Коммит в GitHub после каждой завершённой фичи → автодеплой Vercel

### 9.2. Порядок разработки фич

| №   | Фича                                 | Метод               | Результат                         |
| --- | ------------------------------------ | ------------------- | --------------------------------- |
| 1   | Инициализация проекта                | Plan                | Vite + React + TS, GitHub, Vercel |
| 2   | Simulation Engine (ядро экономики)   | Plan → Tests → Code | Экон. модель + 90% тестов         |
| 3   | ИИ-оппоненты (4 характера, 4 уровня) | Plan → Tests → Code | ИИ-модули + баланс-тесты          |
| 4   | UI: экраны и компоненты              | v0.dev → Code       | Все экраны, графики               |
| 5   | Сохранение, онбординг, справка       | Plan → Tests → Code | localStorage, туториал            |
| 6   | E2E-тесты и балансировка             | Tests → Fix         | Playwright, 100+ прогонов ИИ      |
| 7   | Полировка и релиз                    | Refactor            | Production-деплой Vercel          |

### 9.3. Правила работы с Claude Code

- Каждая задача начинается с команды: «Составь план реализации [X], запиши в PLAN.md»
- После утверждения плана: «Напиши тесты по плану, не пиши реализацию»
- После написания тестов: «Реализуй код, чтобы все тесты прошли»
- После green-тестов: «Сделай коммит и пуш в main»

### 10.2. Скиллы и CLAUDE.md

В корне репозитория должен быть файл `CLAUDE.md` с инструкциями для Claude Code:

- Описание проекта, стека и структуры файлов
- Правила TDD: сначала план, потом тесты, потом код
- Конвенции: именование файлов, структура компонентов, формат коммитов (Conventional Commits)
- Карта экономической модели: формулы, коэффициенты, диапазоны значений
- Список запрещённых паттернов: не использовать `any` в TypeScript, не пропускать тесты, не коммитить красные тесты

### 10.3. Рекомендуемые дополнительные инструменты

- **v0.dev (Vercel)** — генерация UI-компонентов по описанию, экспорт в проект
- **Cursor / Windsurf** — альтернативные ИИ-IDE для параллельной работы с Claude Code
- **ESLint + Prettier** — автоматический контроль качества кода
- **Husky + lint-staged** — пре-коммит хуки: линтинг + тесты перед коммитом

## 11. Требования к тестированию

- Юнит-тесты ядра симуляции и ИИ-модулей: покрытие не менее 90%
- E2E-тесты ключевых сценариев (Playwright)
- Тестирование баланса: автоматический прогон 100+ игр на каждом уровне сложности — проверка winrate игрока
- Тестирование экономики: проверка сходимости, отсутствие эксплойтов, адекватные диапазоны значений
- Кросс-браузерное тестирование на мобильных и десктопе

---

## 12. Критерии приёмки

- Игра запускается в браузере без серверной части и полностью играбельна одним игроком
- 4 уровня сложности ИИ работают корректно и ощутимо отличаются
- Экономическая модель даёт адекватные результаты (12-периодная игра завершается без аномалий)
- Сохранение и загрузка игры работают корректно
- Интерфейс полностью на русском языке
- Адаптивность под мобильные устройства
- Исходный код на GitHub, сайт задеплоен на Vercel и доступен по ссылке
